 # 부하 테스트 리포트: 대기열 상태 조회 API

  ## 1. 개요
  - **테스트 일시:** 2026-04-29
  - **대상 API:** `GET /api/queue/1/status`
  - **테스트 목적:** 대기열 상태 조회 API의 동시
  접속자 수 증가에 따른 응답 성능, 실패율, CPU 사
  용률을 분석하고 단일 인스턴스 기준 임계 구간을
  확인

  ## 2. 테스트 환경
  - **Server:** Localhost (Spring Boot 3.x, Java 17)
  - **Database:** Redis (대기열 관리용)
  - **Tool:** k6
  - **Monitoring:** Windows 작업 관리자, PowerShell CPU 샘플링

  ## 4. 분석

  ### 4.1. 리팩토링 이후 변화
  - `checkQueue()`에서 stale cleanup을 제거하고, `queue-status.lua`를 조회 전용으로 분리한 이후 
  `queue-status` API의 응답 시간이 전반적으로 개선되었다.
  - `700 ~ 1000 VUs` 구간에서는 실패 없이 안정적으로 동작했다.
  - CPU 점유율도 부하 증가에 따라 비교적 완만하게 상승하여, 리팩토링 이후 전반적인 안정화 효과를 확인할 수 있었다.

  ### 4.2. 임계 구간
  - 동일한 `sleep 2s`, `duration 1m` 조건에서 `1000 VUs` 구간부터 간헐적으로 `connect refused`가 재현되기
  시작했다.
  - `1500 VUs` 이상부터는 실패율이 약 `4 ~ 5%` 수준으로 발생하여, 로컬 단일 인스턴스 기준 한계 구간에 진입한 것으로 
  판단된다.

  ### 4.3. 병목 특성
  - `1500 VUs` 구간에서 CPU 최대 사용률은 약 `52%`, `2000 VUs` 구간에서도 약 `71%` 수준이었다.
  - CPU가 완전히 포화되기 전에 실패율이 증가한 점을 고려하면, 현재 병목은 단순 애플리케이션 연산량보다는 다음 요인에 더 
  가깝다.
  - 톰캣 연결 수용 한계
  - 로컬 소켓/커넥션 처리 한계
  - 단일 인스턴스 burst 처리 한계 수준까지는 안정적으로 동작했다.
  - `1000 VUs`부터 불안정 징후가 나타났고, `1500 ~2000 VUs` 구간에서는 실패율 `4 ~ 5%` 수준이 발생했다.
  - 단일 로컬 인스턴스 기준으로는 `1000 ~ 1500VUs` 사이가 임계 구간으로 판단된다.

  ## 6. 향후 계획
  - `enter + status + heartbeat` 혼합 시나리오 부하 테스트 진행
  - `hold`, `reserve` API 부하 테스트 진행
  - 필요 시 `stages` 방식으로 burst 한계와 지속  부하 한계를 분리 측정
  - 향후 다중 인스턴스/로드 밸런싱 환경에서 비교 테스트 진행
