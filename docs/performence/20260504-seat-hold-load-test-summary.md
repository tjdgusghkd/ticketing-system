 # 부하 테스트 리포트: 좌석 선점(Seat Hold) 성능
  개선 및 한계 분석

  ## 1. 개요
  - **테스트 일시:** 2026-05-04
  - **대상 기능:** 좌석 선점 흐름
  - **테스트 시나리오:** `queue enter ->
  heartbeat -> seat hold`
  - **테스트 목적:** 좌석 선점 로직 리팩토링 이후
  응답 성능, 처리량, CPU 사용률 변화를 확인하고
  병목 구간을 파악

  ## 2. 테스트 환경
  - **Server:** Localhost (Spring Boot 3.x, Java
  17)
  - **Database:** MySQL + Redis
  - **Tool:** k6
  - **Monitoring:** Windows 작업 관리자,
  PowerShell CPU 샘플링

  ## 3. 리팩토링 내용
  ### 3.1. 1차 리팩토링
  - `holdSeat()` 내부 Redis 구간을 Lua로 이동
  - 기존 Java의 `GET -> conflict 검사 -> SET ->
  SADD` 흐름을 원자화
  - 좌석 hold 충돌 검사를 Redis Lua 스크립트에서
  처리하도록 변경

  ### 3.2. 2차 리팩토링
  - `cleanupAndCountUserHolds()`를 Lua로 이동
  - `user:hold` 정리와 hold 개수 계산을 Redis Lua
  script로 처리
  - 중간 과정에서 스크립트 경로 오연결 및 Lua 변
  수 오타를 수정

  ## 7. 분석
  ### 7.1. 핵심 개선 포인트
  - 좌석 선점 성능 개선의 핵심은 `holdSeat()`
  Redis 구간을 Lua로 옮긴 1차 리팩토링이었다.
  - 이 변경으로 Redis round-trip 감소와 동시성 처
  리 구간 축소 효과를 확인할 수 있었다.

  ### 7.2. 추가 Lua화 효과
  - `cleanupAndCountUserHolds()` Lua화는 고부하
  구간(150/200 VUs)에서는 일부 개선이 있었지만,
  100 VUs 구간에서는 오히려 악화되었다.
  - 따라서 2차 리팩토링은 일관된 개선 효과로 보기
  는 어렵다.

  ### 7.3. 병목 특성
  - Seat hold는 Queue보다 훨씬 빠르게 응답 지연이
  증가했다.
  - 초기 baseline에서는 `100 VUs`부터 급격한 지연
  이 시작되었다.
  - 1차 리팩토링 이후 한계 구간은 뒤로 밀렸지만,
  고부하 구간에서는 여전히 응답 지연이 크다.
  - 병목은 Redis 처리 비용, hold 정리 비용, DB 조
  회 및 선점 흐름 전체의 결합 비용으로 보는 것이
  타당하다.

  ## 8. 결론
  - 좌석 선점 쪽에서는 `holdSeat()` Redis 구간
  Lua화가 가장 큰 개선 효과를 냈다.
  - 반면 `cleanupAndCountUserHolds()` Lua화는 일
  부 구간에서만 효과가 있었고, 전반적으로는 추가
  보정 수준에 가깝다.
  - 현재 단계에서는 Seat Hold도 1차 리팩토링은 완
  료된 상태로 판단할 수 있다.

  ## 9. 향후 계획

  - 필요 시 hold-only와 mixed 시나리오를 분리해
  재측정
  - DB 조회 최소화 여부 검토
  - 이후 시스템 튜닝 및 확장 구조 검토
