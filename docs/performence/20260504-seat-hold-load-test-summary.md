 # 부하 테스트 리포트: 좌석 선점(Seat Hold)성능 개선 및 한계 분석

  ## 1. 개요
  - **테스트 일시:** 2026-05-05
  - **대상 기능:** 좌석 선점 흐름
  - **테스트 시나리오:** `queue enter ->
  heartbeat -> seat hold`
  - **테스트 목적:** 좌석 선점 로직 리팩토링 이후 응답 성능, 처리량, CPU 사용률 변화를 확인하고 병목 구간을 파악

  ## 2. 테스트 환경
  - **Server:** Localhost (Spring Boot 3.x, Java 17)
  - **Database:** MySQL + Redis
  - **Tool:** k6
  - **Monitoring:** Windows 작업 관리자, PowerShell CPU 샘플링

  ## 3. 리팩토링 내용

  ### 3.1. 1차 리팩토링
  - `holdSeat()` 내부 Redis 구간을 Lua로 이동
  - 기존 Java의 `GET -> conflict 검사 -> SET -> SADD` 흐름을 원자화
  - 좌석 hold 충돌 검사를 Redis Lua 스크립트에서 처리하도록 변경

  ### 3.2. 2차 리팩토링
  - `cleanupAndCountUserHolds()`를 Lua로 이동
  - `user:hold` 정리와 hold 개수 계산을 Redis Lua script로 처리
  - 중간 과정에서 스크립트 경로 오연결 및 Lua 변수 오타를 수정

  ## 7. 추가 튜닝 결과

  ### 7.1. 설정 튜닝
  - 톰캣 설정 조정은 seat hold 흐름 기준 유의미한 개선을 만들지 못했다.
  - HikariCP 설정 조정도 기존 기준보다 응답시간과 처리량 측면에서 개선이 없었다.
  - Redis Lettuce pool 조정 역시 뚜렷한 개선 효과가 확인되지 않았다.

  ### 7.2. DB 인덱스 확인
  - `reservation`, `schedule_seat` 테이블에 인덱스를 추가하고 `EXPLAIN`으로 사용 여부를 확인했다.
  - `reservation` 쿼리는 `ref`, `schedule_seat` 쿼리는 `const` 수준으로 인덱스를 정상 사용하고 있었다.
  - 그러나 부하 테스트 기준으로는 seat hold 성능 개선 효과가 뚜렷하지 않았다.

  ## 8. 분석

  ### 8.1. 핵심 개선 포인트
  - 좌석 선점 성능 개선의 핵심은 `holdSeat()` Redis 구간을 Lua로 옮긴 1차 리팩토링이었다.
  - 이 변경으로 Redis round-trip 감소와 동시성 처리 구간 축소 효과를 확인할 수 있었다.

  ### 8.2. 추가 Lua화 효과
  - `cleanupAndCountUserHolds()` Lua화는 고부하 구간(150/200 VUs)에서는 일부 개선이 있었지만, 100 VUs 구간에서는 오히려 악화되었다.
  - 따라서 2차 리팩토링은 일관된 개선 효과로 보기는 어렵다.

  ### 8.3. 병목 특성
  - Seat hold는 Queue보다 훨씬 빠르게 응답 지연이 증가했다.
  - 초기 baseline에서는 `100 VUs`부터 급격한 지연이 시작되었다.
  - 1차 리팩토링 이후 한계 구간은 뒤로 밀렸지만, 고부하 구간에서는 여전히 응답 지연이 크다.
  - 현재 병목은 인덱스 부족보다는 다음 성격에 더 가깝다.
  - Redis 처리 비용
  - hold 정리 비용
  - DB 조회 및 선점 흐름 전체의 결합 비용
  - 동시성 제어 및 락 대기

  ## 9. 결론
  - 좌석 선점 쪽에서는 `holdSeat()` Redis 구간 Lua화가 가장 큰 개선 효과를 냈다.
  - 반면 `cleanupAndCountUserHolds()` Lua화는 일부 구간에서만 효과가 있었고, 전반적으로는 추가 보정 수준에 가깝다.
  - 설정 튜닝과 DB 인덱스 추가는 기대했던 수준의 개선을 만들지 못했다.
  - 현재 단계에서는 Seat Hold도 1차 리팩토링은 완료된 상태로 판단할 수 있다.

  ## 10. 향후 계획
  - 배포 환경에서 seat hold 흐름 재측정
  - 필요 시 hold-only와 mixed 시나리오를 분리해 재검증
  - reserve는 대규모 반복 부하 테스트보다 기능 및 정합성 검증 중심으로 유지
  - 이후 배포 환경에서 다중 인스턴스 및 확장 구조 검토
