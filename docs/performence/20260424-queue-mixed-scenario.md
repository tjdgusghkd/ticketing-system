 # 부하 테스트 리포트: 대기열 혼합 시나리오 API

  ## 1. 개요
  - **테스트 일시:** 2026-04-24
  - **대상 시나리오:** `POST /api/queue/{scheduleNo}/
  enter` + `GET /api/queue/{scheduleNo}/status` +
  `POST /api/queue/{scheduleNo}/heartbeat`
  - **테스트 목적:** 실제 대기열 사용자 흐름(진입 + 상
  태 조회 + heartbeat) 기준으로 응답 성능과 서버 자원
  점유율(CPU)을 분석

  ## 2. 테스트 환경
  - **Server:** Localhost (Spring Boot 3.x, Java 17)
  - **Database:** Redis (대기열 및 heartbeat 관리용)
  - **Tool:** k6 v1.7.1
  - **Monitoring:** Windows 작업 관리자 (CPU 점유율 체
  크)

  ## 4. 상세 분석 및 인사이트

  ### 4.1. 혼합 시나리오에서의 응답 안정성
  - `enter + status + heartbeat`가 함께 동작하는 실제
  대기열 흐름을 기준으로 테스트한 결과, 모든 구간에서
  **실패율 0.00%**를 유지했다.
  - 동시 사용자 100명 환경에서도 `p95` 응답 시간이 약
  **130ms** 수준으로 유지되어, 대기열 혼합 시나리오에
  서도 안정적인 응답 성능을 확인했다.

  ### 4.2. status 단독 테스트 대비 혼합 시나리오 영향
  - 단순 `status` 조회 API 테스트뿐 아니라, `enter`와
  `heartbeat`가 함께 포함된 상황에서도 응답 성능 저하
  가 크지 않았다.
  - 이는 현재 대기열 구조가 단일 조회 성능뿐 아니라,
  실제 사용자 흐름에 가까운 혼합 요청 상황에서도 충분
  히 견고하게 동작함을 의미한다.

  ### 4.3. CPU 사용률 및 서버 자원 관점
  - 테스트 중 CPU 사용률은 최대 **36%** 수준으로 관찰
  되었으며, 동시 사용자 수를 100명까지 높였음에도 급격
  한 자원 고갈 현상은 보이지 않았다.
  - 현재 구간에서는 CPU보다 Redis I/O 및 애플리케이션
  스레드 처리 구조가 다음 관찰 포인트가 될 가능성이 있
  다.

  ### 4.4. 결과 해석 시 유의사항
  - 본 테스트는 로컬 환경에서 수행되었으며, 단일 토큰
  기반 혹은 제한된 사용자 흐름으로 수행했기 때문에 실
  제 운영 환경과 완전히 동일하다고 보기는 어렵다.
  - 또한 본 테스트는 대기열 혼합 시나리오에 대한 검증
  이며, 실제 예매 API(`hold`, `reserve`)가 포함될 경우
  성능 특성은 달라질 수 있다.

  ---

  ## 5. 결론 및 향후 계획
  - **결론:** 대기열 혼합 시나리오(`enter + status +
  heartbeat`)는 동시 사용자 100명 수준에서도 실패 없이
  안정적으로 동작했으며, 응답 시간과 CPU 사용률 모두
  안정적인 범위를 유지했다.
  - **향후 계획:**
      1. **좌석 선점 API 테스트:** `hold` API를 대상으
  로 동시 선점 경쟁 상황에서의 응답 시간과 충돌 처리
  성능 검증 예정
      2. **예약 API 테스트:** 실제 DB 쓰기가 포함된
  `reserve` API를 대상으로 스트레스 테스트 진행 예정
  간의 성능 특성 확인 예정
